NAME

FVWM::Module::Gtk - FVWM::Module with the GTK+ widget library attached  

SYNOPSIS

  use Gtk;
  use lib `fvwm-perllib dir`;
  use FVWM::Module::Gtk;

  my $module = new FVWM::Module::Gtk;

  init Gtk;
  my $window = new Gtk::Window -toplevel;;
  my $label = new Gtk::Label "Hello, world";
  $window->add($label);
  $window->show_all;

  $module->addHandler(M_CONFIGURE_WINDOW, \&configure_toplevel);
  $module->addHandler(M_CONFIG_INFO, \&some_other_sub);

  $module->eventLoop;
 

DESCRIPTION

The FVWM::Module::Gtk package is a sub-class of FVWM::Module that overloads the methods eventLoop and addDefaultErrorHandler to manage GTK+ objects as well. It also adds a new method openErrorDialog.

This manual page details only those differences. For details on the API itself, see the FVWM::Module manpage.  

METHODS

Only those methods that are not available in FVWM::Module or overloaded ones are covered here:
eventLoop
From outward appearances, this methods operates just as the parent eventLoop does. It is worth mentioning, however, that this version enters into the Gtk->main subroutine, ostensibly not to return.
openErrorDialog
This method creates a dialog box using the GTK+ widgets. The dialog has three buttons labeled ``Close'', ``Close All Errors'' and ``Exit Module''. Selecting the ``Close'' button closes the dialog. ``Close All Errors'' closes all error dialogs that may be open on the screen at that time. ``Exit Module'' terminates your entire module.

Good for debugging a GTK+ based module.

addDefaultErrorHandler
This methods adds a M_ERROR handler to automatically notify you that an error has been reported by FVWM. The M_ERROR handler then calls `openErrorDialog()' with the received error text as a parameter to show it in a window.

CAVEATS

In keeping with the UNIX philosophy, FVWM::Module does not keep you from doing stupid things, as that would also keep you from doing clever things. What this means is that there are several areas with which you can hang your module or even royally confuse your running fvwm process. This is due to flexibility, not bugs.
